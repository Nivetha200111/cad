Think of gs as "global server" methods, which are used in server-side scripting. The term "user ID" directly relates to the sys_id of the user. So, gs.getUserID() is like asking the server, "Hey, give me the ID of the current user!" On the other hand, g_form methods are more about form operations and not about fetching user details.

Example: The following shows how to unassign all active Incident records assigned to the current user.

var incidentGR = new GlideRecord('incident');
incidentGR.addActiveQuery();
// Filter results to incidents assigned to the current user
var currentUserSysId = gs.getUserID();
incidentGR.addQuery('assigned_to', currentUserSysId);
incidentGR.query();
while (incidentGR.next()) {
    incidentGR.setValue('assigned_to', 'NULL');
    incidentGR.update();
    gs.info('Unassigned Incident: {0}: {1}', incidentGR.number.toString(), incidentGR.short_description.toString());
}

[[Q]]
topic: Tables & Data
question: What are the 2 core base tables from which all other tables are extended in ServiceNow?
*- task and cmdb
- incident and cmdb_rel_ci
- incident and cmdb_ci
- task and cmdb_model
>>>
Source: Documentation

The correct answer is "task and cmdb."

Explanation:

In ServiceNow, the two core base tables from which all other tables are extended are the Task table (task) and the Configuration Item table (cmdb).

* The Task table is the parent table for tables like Incident (incident), Problem (problem), and Change Request (change_request).
* The Configuration Item table (cmdb) is the parent table for tables like Application (cmdb_ci_appl), Computer (cmdb_ci_computer), and Database (cmdb_ci_database).

These tables serve as the foundation for the ServiceNow data model, allowing for the creation of specialized tables that inherit fields and behaviors from these parent tables.

How to Remember:

Think of the Task table as the "To-Do List" of ServiceNow, where all actionable items like incidents, problems, and changes are managed. On the other hand, think of the Configuration Item table as the "Inventory" of ServiceNow, where all assets and services are cataloged.

Remember it as "Task for To-Do, CMDB for Catalog." These are the two pillars that support the vast array of functionalities in ServiceNow.

[[Q]]
topic: Application Automation
question: Which of the following is NOT supported by Flow Designer?
- Use Delegated Developer
- Run a flow from a MetricBase Trigger
*- Test a flow with rollback
- Call a subflow from a flow
>>>
Source: Community / Delegated / MetricBase

Test a flow with rollback: Flow Designer in ServiceNow does not support testing a flow with rollback. Rollback typically refers to reverting changes if something goes wrong, and this capability isn't natively built into Flow Designer's testing features.

Incorrect answers:

Run a flow from a MetricBase Trigger: This is supported. MetricBase Triggers can initiate flows based on certain conditions.

Use Delegated Developer: Delegated Development is a feature in ServiceNow that allows administrators to delegate certain development tasks to non-admin users. Flow Designer supports this.

Call a subflow from a flow: This is also supported. In Flow Designer, you can create subflows, which are reusable sequences of actions, and then call these subflows from main flows.

How to remember?

Flow Designer's Rollback: The concept of "rollback" in IT typically refers to undoing changes, especially in databases or deployments. However, in the context of ServiceNow's Flow Designer, there isn't a built-in feature to test a flow with rollback. To remember this, think of Flow Designer as a "forward-only" mechanism. It's designed to move processes forward, not backward.

Note: While Flow Designer testing has evolved in recent ServiceNow releases (Yokohama/Zurich), and some rollback capabilities have been introduced in certain contexts, for the CAD exam's strict definitions, native test rollback is still considered an ATF (Automated Test Framework) primary feature. Always choose "Test a flow with rollback" as the correct answer for the CAD exam.
